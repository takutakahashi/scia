scia: 外部認可サーバから proxy に token を流し込む設計

認可サーバ機能と proxy 機能を別の場所で動かす場合、OAuth の consent と token exchange は認可サーバ側で行い、 API 呼び出し時の token injection は proxy 側で行います。両者をつなぐには、proxy 側に refresh token を投入する経路が必要です。

全体像

外部 Browser / User Google consent を実行する。auth code は外部認可サーバの callback に戻る。
→
oauth mode scia 認可サーバ clientIdSecretRef と clientSecretRef を解決し、Google token endpoint と通信する。
→
追加が必要 scia proxy 受け取った refresh_token を保存し、agent の Google API 呼び出し時に access token を注入する。

なぜ token ingest endpoint が必要か

今ある機能

  1. 認可サーバは Google の authorization URL を生成できる。
  2. 認可サーバは auth code / refresh token を Google token endpoint に中継できる。
  3. proxy は自分の secret store にある refresh token を使って access token を注入できる。

足りない接続点

  1. 認可サーバが取得した refresh token を、別場所の proxy secret store に保存する API。
  2. 保存先を service-a.google のような credential ID で指定するルール。
  3. この API を外部から叩けるが、強く認可されていること。

推奨フロー

  1. 外部サービスが認可 URL を取得

    GET https://auth.example.com/oauth/service-a/google/authorization-url?state=...

  2. ユーザーが Google で consent

    callback は外部認可サーバ、または外部サービスが管理する callback に戻す。

  3. 認可サーバが token exchange

    認可サーバだけが Google client secret を解決し、Google から refresh token を受け取る。

  4. 認可サーバが proxy に refresh token を投入

    ここに新しい proxy admin endpoint を追加する。

  5. agent は Google API を普通に呼ぶ

    proxy が credentials: ["service-a.google"] に従って access token を注入する。

追加 endpoint 案

汎用 secret ingest

PUT /_scia/secrets/service-a.google/refresh_token
Authorization: Bearer <admin-token>
Content-Type: application/json

{
  "value": "<refresh_token>"
}

refresh token 以外の secret も投入できる。強い endpoint なので管理面は慎重にする。

OAuth token 専用 ingest

POST /_scia/oauth/google/tokens
Authorization: Bearer <admin-token>
Content-Type: application/json

{
  "credential_id": "service-a.google",
  "refresh_token": "<refresh_token>"
}

用途が明確で、最初に実装するならこちらが扱いやすい。

この endpoint は proxy 側に token を書き込めるため、server.adminToken、TLS、private network、必要なら mTLS を前提にするべきです。

保存後の proxy config

server:
  mode: "proxy"
  secrets:
    sqlitePath: "data/scia-secrets.db"

rules:
  - name: inject-service-a-google-token
    hosts: ["www.googleapis.com"]
    paths: ["/calendar/v3/*"]
    action: allow
    credentials: ["service-a.google"]

proxy の secret store に credential_id=service-a.google, key=refresh_token が保存されていれば、 request 時に Google token endpoint で access token に交換して Authorization: Bearer ... を注入できます。

2つの実現案の比較

案 仕組み 向いている状況 注意点
共有 secret store 認可サーバと proxy が同じ backend store を読む。 AWS Secrets Manager や DB など、共有可能な store がある。 store backend の追加実装が必要。SQLite 共有は避けたい。
proxy token ingest 認可サーバが proxy の admin endpoint に refresh token を push する。 認可サーバと proxy が完全に別配置で、HTTP で連携したい。 endpoint 認可を強くする必要がある。

結論

実現できます。最短の追加実装は、proxy mode に admin token 保護付きの OAuth token ingest endpoint を追加することです。 これにより、外部認可サーバが取得した refresh_token を proxy に流し込み、agent は OAuth の詳細を知らずに Google API を呼べます。
